# Git Commit — 项目提交工作流

当用户说出以下触发词时，自动执行本工作流：
- "帮我提交git" / "提交git" / "git提交" / "提交一下"
- "commit" / "git commit"（在中文语境下）

---

## 工作流（严格按顺序执行，不可跳过任何步骤）

### 步骤 1：查看变更

```bash
git status
git diff --stat
git log --oneline -5
```

### 步骤 2：确认变更范围

在执行步骤 1 的同时，读取 `git diff` 原文，确认：
- 新增文件（untracked）
- 修改文件（modified）
- 删除文件（deleted）
- 重命名（renamed）

用表格向用户展示变更摘要，格式固定为：

```
| 操作 | 文件 | 说明 |
|------|------|------|
| 新增 | path/to/file | 简要说明 |
| 修改 | path/to/file | 简要说明 |
| 删除 | path/to/file | 简要说明 |
| 重命名 | old → new | 简要说明 |
```

### 步骤 3：撰写提交消息（严格按模板）

**只写一版，不做选择题。** 始终使用以下详细模板：

---

#### 提交消息模板（死板格式，每次保持一致）

```
<type>: <一句话概括>

- <文件/模块分类1>
  · <具体变更>（如有数据变化，标注 旧值 → 新值，附简短理由）
  · <具体变更>
- <文件/模块分类2>
  · <具体变更>
  · <具体变更>
```

---

#### 模板规则（必须严格遵守）

1. **前缀**：使用约定式提交前缀 `feat:` / `fix:` / `refactor:` / `chore:` / `docs:`
2. **第一行**：中文，一句话概括，不超过 40 字
3. **空行**：第一行和变更列表之间必须空一行
4. **缩进**：一级用 `- `（破折号+空格），二级用 `  · `（两个空格+中间点+空格）
5. **数据变化**：任何数字变化必须标注 `旧值 → 新值`，并在括号中附简短理由
6. **文件路径**：始终用完整相对路径，如 `英伟达/260509-01_整体分析.md`
7. **顺序**：先项目级变更（目录结构、配置文件），再按公司字母序排列业务变更
8. **分组**：变更超过 10 行时，用空行分隔不同大类

### 步骤 4：提交

1. `git add` 精确指定每个变更文件（不用 `git add .`）
2. `git status` 确认暂存区正确
3. `git commit -m "消息"`
4. `git status` 确认工作区干净

### 步骤 5：提交后立即推送

提交成功后，**自动执行** `git push`。不询问用户。

```bash
git push
```

推送完成后，报告用户：
- 提交 hash
- 推送状态
- 工作区状态

## 写作铁律

| 规则 | 说明 |
|------|------|
| **只写一版** | 详细版，不做选择题 |
| **格式死板** | 严格遵循模板的缩进、符号、空行 |
| **数据必标** | 任何数字变化必须 `旧值 → 新值` |
| **理由必有** | 评分变化必须括号附因 |
| **路径完整** | 始终写完整相对路径 |
| **排序固定** | 项目级在前，公司按字母序在后 |
| **提交即推送** | `git commit` 后自动 `git push`，不询问 |
